# Qiita 記事作成

技術記事を新規作成し、限定共有（`private: true`）の状態で完成させる。

記事の品質は、書き始める前の設計で決まる。テーマを受け取ったらすぐに書き始めず、まず核を固める。

## Step 1: 核を固める

ユーザーと対話して以下を明確にする。すべて埋まるまで執筆に入らない。

| 項目 | 問い | 悪い例 | 良い例 |
|---|---|---|---|
| 主張 | この記事で言い切ることは何か？ | 「Terraform は便利」 | 「Control Tower の OU 未登録エラーは AFT のドキュメントに書いていない。こう回避する」 |
| 読者 | 誰が、どういう状況で読むか？ | 「エンジニア全般」 | 「AWS Organization を初めて AFT で管理しようとして詰まった人」 |
| 差分 | 既存記事と何が違うか？ | 「まとめ記事」 | 「公式ドキュメントにない回避策と、なぜそうなるかの内部挙動の説明」 |
| スコープ | 何を書かないか？ | （未定義） | 「Control Tower 自体の導入手順は扱わない」 |

核が曖昧なまま書くと、AI っぽい総論記事になる。ユーザーが「とりあえず書いて」と言っても、最低限「主張」と「読者」は確認する。

## Step 2: 構成を決める

核をもとに、見出し構成（H2 レベル）を箇条書きで提示する。

守ること:
- 見出しは名詞句にする。「○○は△△だ」という命題型にしない
- 「はじめに」「まとめ」は不要。冒頭は本題から入り、末尾は最後の節で自然に閉じる
- 読者の問いの順に並べる。書き手の調査順ではなく、読み手が知りたい順

構成をユーザーに確認し、合意を得てから執筆に入る。

## Step 3: ファイル作成

```bash
pnpm new <basename>
```

- `<basename>` は記事タイトルの要約を日本語含む kebab-case で指定する（例: `terraformでaws-control-tower-account-factoryを使ったら-ou-is-not`）
- 生成されるファイル: `public/YYYY-MM-DD_<basename>.md`（日付は自動付与、`private: true` で作成される）
- `organization_url_name` と `private` はこの段階では変更しない

### タイトル

- 読者が検索しそうなキーワードを含める
- 30〜50 字程度。短すぎると内容が伝わらず、長すぎると検索結果で切れる
- 主張か状況を具体的に書く。「〜について」「〜のまとめ」で終わらせない
- 悪い例:「Terraform のエラー対処」→ 良い例:「AFT で OU 未登録エラーが出たときの回避策」

### タグ

- 技術名・ツール名を優先する（例: `Terraform`, `AWS`, `ControlTower`）
- 「初心者」「入門」のような属性タグは、記事の主な読者層に合致する場合のみ
- 最大 5 つ。多すぎると焦点がぼやける
- Qiita で既存のタグ名に合わせる（表記揺れを避ける）

## Step 4: 執筆

`/stop-ai-slop-jp` を適用して執筆する。

### 書く前に意識すること

- **私の経験から書く**: 「○○について解説します」ではなく、「○○をやったらこうなった」から入る
- **具体を先に、抽象は後から**: コード・コマンド・エラーメッセージを先に見せて、それから説明する
- **読者の時間を尊重する**: 前提知識の説明は最小限にし、公式ドキュメントへのリンクで済ませる

### 文体

- **ですます調で統一する**。ただし「です」「ます」の単調な繰り返しを避け、「でした」「でしょう」「ません」「ですが」「ました」などで変化をつける
- **主語は「私」で統一する**。「我々」「筆者」「本稿では」を使わない
- **事実だけを断定する**。自分で確認した事実は言い切る。推測・伝聞・未検証の情報には「らしいです」「ようです」「と思います」を残し、根拠なく断定しない
- stop-ai-slop-jp の採点で 35/50 以上を目指す

### 段落の書き方

- 一文一改行。段落の区切りは空行
- 一つの段落には一つのトピックだけを置く
- 段落の最初の文を読めば、何の話かわかるようにする
- 同じ主張を言い換えて繰り返さない。一つの主張は一度だけ書く
- 因果を書くときは、なぜそうなるかの機構を一文で示す。「A だと B になる」だけで終わらせない

### コードブロック

- 実行可能なコードを載せる。擬似コードや省略コードは避ける
- ファイルパスや言語をコードブロックのメタデータに指定する
- 長いコードは分割して、各ブロックの前に何をしているか一文で説明する

### 外部ページへの参照リンク

公式ドキュメントなど外部ページのリンクは、`<>` で囲まず、前後に空行を挟んだ裸の URL（bare URL）で書く。

```text
（前の段落）

https://example.com/docs

（次の段落）
```

Qiita は単独行の裸 URL をカード表示するが、`<>` で囲むとカードにならない。VSCode の markdownlint（MD034）が保存時に `<>` を付ける場合は、`.vscode/settings.json` の `markdownlint.config` で MD034 を無効化する（`.vscode` は gitignore 対象のためローカル設定）。

## Step 5: セルフレビュー

執筆後、以下の観点で点検する。

### 核との整合

- [ ] Step 1 で決めた主張が記事内で言い切られているか
- [ ] スコープ外の話題に脱線していないか
- [ ] 読者が設定した状況の人にとって、過不足ない内容か

### 文体

- [ ] ですます調で統一されているか（「だ・である」が混在していないか）
- [ ] 「です」「ます」だけの単調な文末になっていないか
- [ ] 主語が「私」で統一されているか（「我々」「筆者」が混入していないか）
- [ ] 事実以外を断定していないか（推測・伝聞に適切な表現が残っているか）

### stop-ai-slop-jp チェック

- [ ] false agency がないか（モノが主語で人間の動詞をやっていないか）
- [ ] 反証可能な具体的主張があるか
- [ ] 中間温度（「悪くない」「微妙」）が出てくるか
- [ ] AI 偏愛語（解像度、本質、手触り、熱量）を撒いていないか
- [ ] 命題型 H2 がないか
- [ ] 両論併記で判断を放棄していないか

### 構成と段落

- [ ] 一つの段落に一つのトピックだけが置かれているか
- [ ] 同じ主張を言い換えて繰り返していないか
- [ ] 因果の機構（なぜそうなるか）が省略されていないか

問題があれば修正し、ユーザーに完成版を提示する。

## Step 6: 完了報告

記事完成後、以下を出力する:

```
## 記事作成完了

- **ファイル**: public/<basename>.md
- **タイトル**: <title>
- **ステータス**: 限定共有（private: true）

### 次のステップ
1. 内容を確認し、push してください
2. CI が記事を Qiita に反映します（限定共有として）
3. 承認に出す場合は `/qiita-approve` で申請情報を出力できます
```
