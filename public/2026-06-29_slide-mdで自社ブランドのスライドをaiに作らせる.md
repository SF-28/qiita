AI にスライドを作らせたくて、`SLIDE.md` という Claude Code 向けのスキルパッケージを試しました。
使ってみて感触が良かったので、装飾画像を埋め込むところまで自分で拡張して使っています。

`SLIDE.md` は、スライドのデザインを Markdown の設計書として書き、その設計書を AI に渡してスライドを生成させる仕組みです。
配布元はここにあります。

https://github.com/sho-ai-magic/SLIDE.md

素の状態でも一通りスライドは作れます。
ただ、私が使いたかったテンプレートにはロゴや装飾画像が必須で、そこは標準では埋められませんでした。
この記事では、素の使い心地を先に書いてから、私が足した 2 つの工夫（装飾画像の埋め込みと、参考デザインへの寄せ方）を、実際の設定ファイルつきで紹介します。

## SLIDE.md で最初に作ったスライド

最初に作ったのは、`SLIDE.md` 自体を紹介する 7 枚のスライドでした。
やることは、Claude Code にスキルを順番に呼ばせるだけです。

<!-- 画像①: `SLIDE.md` で最初に作った slides 紹介デッキの表紙（元画像 article-images/2026-06-29_slide-md/01-slide-sample-cover.png）をここに挿入 -->

| 手順 | スキル | やること |
|---|---|---|
| 1 | `slide-md-creator` | 参考デザインから配色・フォント・余白を読み取り、デザイン定義`SLIDE-md`を作る |
| 2 | `slide-pattern-creator` | スライド画像からレイアウトを抽出して、再利用できるパターンにする |
| 3 | `slide-deck-builder` | プレゼン内容を流し込み、デザインとパターンを 1 ファイルに束ねた設計書`SLIDE-DECK`を組む |

できあがる `SLIDE-DECK` は、デザイン定義とレイアウトとプレゼン内容が 1 枚に詰まった設計書です。
配布元の README では、このファイルを Claude Design などの AI ツールに渡してスライドを描かせます。
私は手元で HTML を出したかったので、設計書から HTML スライドを生成する小さなスキル(`slide-builder`)を`skill-creator`で追加しました。

## デザイン定義を一度書けば使い回せる仕組み

使ってみて一番効いたのは、デザイン定義とレイアウトが分かれていることです。

<!-- 画像②: `SLIDE-md`+SLIDE-PATTERN → SLIDE-DECK → SLIDE の 3 層構造（元画像 article-images/2026-06-29_slide-md/02-slide-sample-3layers.png）をここに挿入 -->

`SLIDE-md` には配色やフォント、余白に加えて、全ページ共通の枠（上部のタイトル線、右下のページ番号、装飾の位置）を書きます。
一方でレイアウトパターンは、本文エリアの構造だけを持ちます。
枠を `SLIDE-md` 側に集約してあるので、どのパターンを選んでも枠の見た目が揃います。これが、スライドごとにデザインがバラつかない理由でした。

おかげで、デザイン定義を 1 回書いてしまえば、別のプレゼンを作るときは内容を流し込むだけで同じ見た目になります。

## 装飾画像を埋め込む Assets セクションの追加

ここからが工夫の一つ目です。

素の `SLIDE-md` は、配色やフォント、余白、枠までは持てますが、ロゴや装飾画像といった assets を置く場所がありませんでした。

拡張というと大げさですが、`SLIDE-md` が Markdown だったので、対応は簡単でした。
`## Assets` というセクションを足し、スライドの種類ごとに「どの画像を、どこに、どのサイズで置くか」を宣言するだけです。

```markdown:SLIDE-md

## Assets

| スライドタイプ | 右上 | 左下 | 右下 | その他 |
|---|---|---|---|---|
| title（表紙） | title-top-right.png (180px) | title-bottom-left.png (180px) | なし | oec-logo.png（中央下） |
| section（見出し） | section-top-right.png (80px) | section-bottom-left.png (120px) | なし | なし |
| body（本文） | body-top-right.png (80px) | body-tagline.png | body-page-ring.png (40px) | なし |
| closing | なし | なし | なし | closing-left.png + closing-right.png |

```

画像ファイル名は `{スライドタイプ}-{位置}.png` で統一しました。
`title-top-right.png` を見れば、表紙の右上に置く画像だと名前だけで分かります。
位置をファイル名に持たせておくと、後でどの画像がどこに対応するかを探さずに済みます。

この `## Assets` を読めるように、スキルも手を入れました。

- `slide-md-creator`: 参考スライドから装飾画像を切り出して `assets/` に集め、`## Assets` を書く
- `slide-deck-builder`: 設計書に `## Assets` と画像フォルダへの相対パスを埋め込む
- HTML 生成の段階: スライドごとの `frame`（title / section / body / closing）を見て、対応する画像を `<img>` で絶対配置する

肝は、各スライドに `frame` という 1 行を持たせたことです。
`frame: title` と書いたスライドには表紙用の装飾が、`frame: body` には本文用の装飾が自動で乗ります。
CSS で枠を毎回描くのではなく実画像を相対パスで参照させているので、ロゴの細部までそのまま出ます。

## 参考 PDF にデザインを寄せる AI セルフレビュー

工夫の二つ目は、生成したスライドをベーステンプレートに寄せるやり方です。

最初に生成したスライドは、配色こそ合っていましたが、装飾の位置とサイズがずれていました。
右上のアイコンが少し大きい、ロゴの下マージンが足りない、といった細かいずれです。
私が目で見て「もう少し左」「少し小さく」と指示しても、なかなか合いませんでした。

効いたのは、私が差分を指摘するのをやめて、AI 自身に比較させたことです。
ベーステンプレート PDF と、生成したスライドのスクリーンショットを並べて渡し、「この 2 つの装飾の位置とサイズの違いを挙げて直して」と頼みます。
スクリーンショットは Playwright で撮らせました。
これを何度か繰り返すと、ピクセル単位のずれが少しずつ埋まっていきました。

最終的に、デザイン確認用の `sample.html` は参考 PDF とほぼ重なる位置まで寄りました。
ここまで合わせておくと、本番のスライド生成でも `sample.html` の枠の CSS をそのまま使い回せます。


## 生成できた自社ブランドのスライド

最終的に、ゴールデンアンバーとパステルを基調にしたデザインで、表紙・セクション・本文・クロージングそれぞれに装飾が入ったスライドが生成できました。
表紙には四隅の装飾と中央のロゴ、本文には右下のページ番号リングが、どのスライドでも同じ位置に乗ります。

<!-- 画像④: 本文スライド。上線・右上アイコン・左下タグライン・右下のページ番号リングが同じ位置に乗る（元画像 article-images/2026-06-29_slide-md/04-oec-body.png）をここに挿入 -->

この作り方が向くのは、決まったテンプレートを何度も再現したい場合だと思います。
一度デザイン定義と装飾画像を整えてしまえば、次からは内容を流し込むだけで同じ見た目のスライドが出てきます。
逆に、一回きりの資料を作るだけなら、ここまで設計書を整える手間は要りません。普通にスライドツールで作ったほうが速いでしょう。
